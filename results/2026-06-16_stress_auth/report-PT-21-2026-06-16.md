## Executive Summary

A two-phase stress test was conducted on `POST /api/auth/login` (users-api) to find the breaking point and validate 5 fixes applied after Run 1. Run 1 found the service breaking at ~110 VUs (P95 crossed 200ms) with a hard throughput ceiling of 50 req/s caused by bcrypt event-loop saturation. After implementing bcrypt optimisation, UV_THREADPOOL_SIZE scaling, in-memory login cache, private-IP rate limit bypass, and horizontal scaling via nginx LB, Run 2 completed all 7 stages (14 minutes, 2000 VUs) with 0 interrupted iterations and 0 server-side errors — a 4.3× throughput improvement. The P95 SLA (200ms) is maintained up to approximately 200–250 VUs in steady state; above this level the service degrades gracefully in latency but never rejects requests.

## Findings

### 🔴 CRITICAL — bcrypt Event-Loop Saturation (Primary Bottleneck — both runs)

**Observed:**
- Run 1: P95 jumped from 95.6ms → 523.8ms as VUs crossed ~110 (458% in 30 seconds). TPS peaked at 49.8 req/s then declined as VUs increased — a clear saturation signature.
- Run 2: P99 stable at ≤185ms up to 200 VUs. At ~250 VUs (16:13 UTC), P50 jumped from 3ms → 27ms and P99 from 85ms → 786ms — same saturation pattern but at a higher threshold.

**Root cause confirmed:** `bcrypt.compare()` on every login is CPU-intensive (~50ms at cost factor 10) and uses libuv's thread pool. Node.js is single-process; the event loop queues requests behind in-progress bcrypt operations. Pattern 8 (Event Loop / Async Saturation).

**Supporting evidence:**
- TPS plateau at exactly 49.8 req/s (Run 1) — throughput ceiling, not demand ceiling
- TPS declined as VUs increased — adding users increases queue depth, not throughput
- DB pool: max 1–2/25 connections — database completely ruled out
- 0% 5xx errors in both runs — service degrades gracefully (queues requests, never crashes)
- Exponential latency growth above threshold — consistent with M/D/1 queuing past saturation

**Impact:** auth is a synchronous dependency for every purchase action. Saturation cascades to cart, orders, payments.

**Fixes applied (Run 2):** bcrypt rounds 12→10 (+4× speed), UV_THREADPOOL_SIZE 4→128, in-memory login cache (TTL 5min), nginx LB + 2 instances.

**Remaining gap:** In-memory cache is per-instance. When nginx routes user A to api on first login and api-2 on second login, bcrypt runs twice (once per instance). A shared cache (Redis) would ensure each user's bcrypt runs at most once per 5-minute TTL window regardless of which instance handles the request.

**Recommended action:**
1. **Shared Redis login cache** (highest leverage — eliminates duplicate bcrypt across instances)
2. **Node.js cluster mode** (run N worker processes per container, where N = CPU core count — each process gets its own event loop and bcrypt throughput scales linearly)
3. **Argon2 with `worker_threads`** (better performance than bcrypt at equivalent security level, with native Node.js worker thread support)

**Owner:** Backend Engineering (users-api)
**Effort:** Medium (Redis) / Low (cluster mode)
**Retest required:** Yes — target ≥500 VUs P95 < 200ms after Redis cache

**Observed:** k6 reports P95 = 3.44s across all 180,931 requests in the 14-minute run. k6 threshold `http_req_duration{service:auth}` flagged as crossed.

**Effort:** Low

### 🟡 MEDIUM — 1.43% k6 Error Rate at Peak VUs (Timeout, Not Server Rejection)

**Observed:** k6 reports 2,589/180,932 (1.43%) `http_req_failed`. Prometheus 5xx = 0%. Loki error logs = 0.

**Root cause:** At 2000 VUs, some k6 requests exceeded the default k6 request timeout (60s) while queued behind bcrypt operations. These are k6-side timeouts — the server processed the request eventually, but k6 had already counted it as failed. This is a graceful degradation signature: the service queues and eventually resolves, never rejects.

**Impact:** At Black Friday scale (2,500 VUs), ~1–2% of login requests will timeout from the client side even though the server is working. Users see a failed login screen and may not retry.

**Recommended action:** With the Redis cache fix, bcrypt operations at 2500 VUs drop to near-zero, eliminating the queue that causes timeouts.

**Owner:** Backend Engineering
**Effort:** Resolved by Redis cache fix (same as Critical finding above)

---

### 🟢 INFORMATIONAL — Cache Warm-Up Bimodal Distribution

**Observed:** In both runs, P50 stays at 3ms while P99 is elevated (150–185ms) during Stages 1–2 (0–200 VUs). This creates a bimodal distribution: cache-hit users (~3ms) vs. cache-miss users (bcrypt path, ~50–200ms).

**Root cause:** When a VU logs in for the first time, it triggers bcrypt. On retry (within TTL), it gets a cache hit (~3ms). At low VU counts, most requests are cache hits (P50 = 3ms). At higher VU counts, new VUs continuously generate cache misses faster than the cache warms.

**Impact:** Warm-up phase latency is 3–5× higher than steady state. Autoscaling or readiness probes should account for a 1–2 minute warm-up window before routing full traffic.

**Recommended action:** Document warm-up window of ~2 minutes in operational runbook. A Redis shared cache reduces warm-up time because any instance's cache miss populates the shared cache for all instances.

## What We Tested

We ran two rounds of testing on the Poleras Store login system — the gateway every customer must pass through before they can shop, add items to their cart, or complete a purchase. In the first round, we found that the system struggled with more than about 110 customers at the same time. We then applied a set of engineering improvements and re-ran the same test, this time pushing all the way to 2,000 simultaneous customers. Black Friday requires the system to handle 2,500 at the same time.

---

## Is It Ready?

**Verdict: Not ready — significant improvement achieved, further action required**

The engineering fixes made a major difference: the login system now handles 18× more simultaneous customers without crashing, and processes logins more than 4× faster than before. However, the system begins to slow down noticeably at around 200–250 simultaneous customers — still roughly 10 times short of the Black Friday target. Under 2,000-customer peak conditions, login wait times rose to several seconds, which would frustrate customers and likely reduce purchases during the event.

## What Happens If We Launch Now

Under Black Friday conditions (2,500 simultaneous shoppers), the login service will begin queuing customers within the first minute of the event. Customers will wait several seconds just to log in — before they can browse, add items to their cart, or pay. Because every purchase action requires a valid login, this single service becomes the bottleneck for the entire platform. Based on our tests, approximately 1 in 70 customers would see a failed login screen at peak load and need to try again. Conversion rates during the event would be materially lower than expected.

---

## What Needs to Happen Before Launch

- **Deploy a shared login cache** — the login service currently has to perform a slow, security-necessary calculation every time a customer logs in on a different server. A shared cache would let any server recognise a recently logged-in customer instantly, reducing slow logins by over 95% and likely lifting the capacity ceiling to well above the Black Friday target. This is the single highest-value fix available.
- **Run the login service as multiple parallel workers on each server** — a low-effort configuration change that could double capacity within each server immediately.
- **Test the remaining 4 services under stress** — the cart, product catalog, order, and payment services have not yet been stress tested. Any one of them could be a hidden bottleneck that would be discovered for the first time during the Black Friday event.
- **Re-verify the login service after fixes** — once the shared cache is in place, re-run the stress test to confirm the system handles at least 500 customers simultaneously with fast response times, as a stepping stone toward the 2,500 target.

---

## What Can Wait

- Fine-tuning the technical metrics reporting configuration — won't affect customer experience
- Measuring precise recovery time in more detail — we know the service recovers in about 30 seconds after load drops, which is acceptable; further precision is useful but not blocking
