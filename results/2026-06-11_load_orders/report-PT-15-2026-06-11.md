### Findings

#### CRITICAL — Finding 1: DB connection pool exhaustion under 100 VU load (Run 1)

**Observed:**  
At exactly t+6:00 (14:01 UTC), the service suffered a complete throughput collapse:
- TPS dropped from **53 req/s → 0.09 req/s** (Prometheus: `sum(rate(http_requests_total{job="orders-service"}[1m]))`)
- `orders_created_total` counter froze at **4,761** — no new orders after 14:01 UTC
- P95 peaked at **133ms** at 14:01 UTC immediately before collapse (Grafana range query)
- k6 recorded a burst of ~100 simultaneous timeout errors — all firing at t+5m30s
- Global error rate reached **2.70%** on orders-service endpoints

**Root cause:**  
Two compounding defects:

1. **Pool undersized:** `pg` pool defaulted to `max: 10` with `connectionTimeoutMillis: 0` (infinite wait). Under 100 VUs, 90 requests simultaneously queued for a connection — indefinitely.
2. **Connection held during external HTTP call:** `pool.connect()` was called at the top of the handler, *before* the cart-service HTTP fetch (~50–100ms). This meant each of the 10 connections was idle-held during the network call, reducing effective pool capacity to near zero under concurrency.

Combined effect: 10 connections × one held per VU during cart fetch = pool exhausted within seconds of reaching full load. The 90 queued VUs waited exactly 60 seconds (k6's default request timeout) before all failing simultaneously — creating the characteristic burst error pattern.

**Evidence:**
- Prometheus TPS chart: `sum(rate(http_requests_total{job="orders-service"}[1m]))` — 53 req/s → 0.09 req/s at 14:01 UTC
- `orders_created_total`: counter value `4761` frozen from 14:01 UTC through end of window
- Loki (`{container="orders-service"}`): last "Order created" log at 13:58 UTC (order POL-2026-02178, traceId: `e3cd580a7dec018cf90e72ce86968404`) — then 4 minutes of silence except `/metrics` scrapes
- k6 log: simultaneous ~100 `request timeout` errors at t+5m30s
- **Loki shows zero error-level logs** — pool exhaustion is silent at the application layer; requests queue inside the Node.js `pg` library before reaching any application code

**Tempo investigation:**  
No trace IDs to investigate. The failure mode never reached application-level spans — requests blocked in the connection pool queue, which generates no OpenTelemetry traces. This is expected behaviour for pool starvation.

**Fix applied:**
- `db.js`: `max: 10 → 25`, `min: 5`, `connectionTimeoutMillis: 5000` (fail-fast instead of infinite queue)
- `server.js`: `pool.connect()` moved *after* cart-service HTTP fetch; `client.release()` called immediately after `COMMIT` (not in `finally`)

**Retest required:** ✅ Completed — Run 2 validates the fix.

## Part 2 — Business Report

### What Was Tested

We simulated 100 simultaneous users creating orders on the Poleras Store platform, representing the expected peak traffic volume during the upcoming Black Friday event. The test ran for 10 minutes and covered the complete purchase flow: user login, cart management, order creation, and order retrieval. The test was executed twice on 2026-06-11.

---

### Key Question: Is It Ready?

**Overall verdict: ✅ Ready — with the fixes already applied**

The first attempt revealed a critical infrastructure defect that would have caused the order creation system to completely stop processing purchases under peak Black Friday load. That defect has been identified, fixed, and re-tested — the second run completed without a single failed order.

---

### What Happened in Each Run

**First attempt (failed):**  
Approximately 5.5 minutes into the test, the order creation system stopped processing entirely. For the remaining 4+ minutes, no orders could be created. The root cause was a configuration limit that allowed only 10 simultaneous database operations — far too few for 100 concurrent shoppers. When that limit was hit, all additional requests silently waited in a queue until they expired. This would have meant **every user attempting to place an order during a Black Friday surge would receive a timeout** after waiting 60 seconds — effectively losing the sale.

**Second attempt (passed):**  
After increasing the database connection limit and optimising how the service holds database resources, the system handled 100 simultaneous users for the full 10-minute test with **zero failed order creations**. Response times were fast and stable throughout, never approaching the acceptable limit.
