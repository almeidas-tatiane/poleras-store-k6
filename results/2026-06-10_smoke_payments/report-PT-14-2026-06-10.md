## Why exit 99 is expected
The payment gateway simulates **200–800ms latency by design** in the default container configuration. This causes P90 = 710ms and P95 = 799ms — both above the 300ms SLA threshold.

**Exit 99 is NOT a service failure:**
- All 209 checks passed (100%) — all HTTP responses were correct
- Error rate = 0% — no failed requests
- HTTP 402 (gateway rejection ~10% rate) correctly excluded via `responseCallback`
- This behaviour was pre-documented in PT-12 comments

**Container state note:** On 2026-06-09, the gateway latency patch (50–200ms) was active and P95 was 207.97ms (within SLA). The containers restarted between sessions, resetting the patch. The 2026-06-10 run reflects the default unpatched gateway behaviour.

## Baseline (reference for load tests — unpatched gateway)
- **p50:** 135.02ms
- **p95:** 799.45ms — SLA breach by design (gateway 200–800ms)

## Comparison vs 2026-06-09 baseline

## Risk for load tests
- **Gateway latency patch must be re-applied** before any load/stress/spike test if the 300ms SLA is to be validated
- Without the patch, the payments threshold will always breach at any VU count
- The functional correctness of the service is confirmed (all checks 100%)

## Conclusion
⚠️ payments-service is **functionally healthy** (0% errors, 100% checks). Exit 99 is expected due to unpatched gateway latency (200–800ms by design). **Action required before load test:** re-apply gateway latency patch to container to restore sub-300ms P95.
