## Findings

### HIGH — P95 Latency SLA Breached Above ~1,800 VUs; Breaking Point Not Yet at 2,500 VU Target

**Observed:**
The k6 global P95 was 1.57s (FAIL). Prometheus shows P95 for successful 200 responses reached 89–93ms at ~2,000 VUs — just within SLA for successful requests. Errors (0.34%) appeared primarily at peak 2,000 VUs with 500-response P95 of ~2–4s.

**Progress vs prior runs:** Breaking point improved from ~1,600 VUs (Run 3) to ~1,800–2,000 VUs (Run 4). However, the Black Friday target is 2,500 VUs — still 25–38% above the breaking point.

**Root cause:** 8 workers handle ~250 VU/worker at peak. At 2,000 VUs, this is at the threshold where event loop lag starts spiking (161ms at ~1,867 VUs). The spike is transient (recovers in 90s) but causes queuing bursts that produce 500 errors and inflated P95.

**Recommended action:** Increase `NODE_CLUSTER_WORKERS` to 12 (or configure autoscaling). At 2,500 VUs / 12 workers = ~208 VU/worker, well below the ~250 VU saturation threshold observed in this run. Alternative: add request queuing/backpressure with 503 (retry-able) instead of 500 (non-retry-able) to prevent cascading failures at peak.

---

### MEDIUM — 931 HTTP Errors (500) at Peak 2,000 VUs

**Observed:**
931 out of 270,103 requests returned HTTP 500 (0.34% error rate — within 0.5% SLA). Errors distributed across all 3 endpoints.

**Compared to Run 3:** Run 3 had 4 errors (all on `/api/categories`). Run 4 has 931 errors spread across all endpoints. The increase is due to running at 2,000 VUs with 8 workers instead of 4 — each worker still saturates at peak, but the saturation is delayed.

**Root cause hypothesis:** Same as Run 3 pattern — at peak VU count, the combined load on Redis and DB connection establishment exceeds per-worker capacity, causing timeout errors. With 8 workers, this threshold is higher but still reached at 2,000 VUs.

**Recommended action:** Apply the backpressure/queue improvement above. Additionally, investigate whether the 500 errors correlate with Redis TTL expiry boundaries (thundering herd) — if so, the P2 TTL jitter fix from Run 3 would also reduce Run 4 errors.

---

### INFORMATIONAL — Event Loop Lag: Transient Spike, Full Recovery

**Observed:**
`nodejs_event_loop_lag_p95_seconds` spiked from 50ms to 162ms between t+11m46s and t+12m16s (~1,867–2,000 VUs). Recovered to 35ms by t+13m16s (~1,000 VUs ramp-down) and to baseline 21ms by t+13m46s.

**Significance:** Unlike Run 3 (4 workers) where the event loop was hypothesized to saturate non-linearly above ~400 VU/worker, the 8-worker configuration shows a transient spike (not sustained saturation) at ~250 VU/worker. The 90s recovery time means the service is self-correcting, but the spike itself causes the error burst.

**For Black Friday planning:** At 2,500 VUs / 12 workers = 208 VU/worker, the event loop lag should remain below 100ms based on the trend in this test (69ms at ~1,100 VUs, 50ms at ~1,700 VUs, spike only at ~250 VU/worker threshold).
