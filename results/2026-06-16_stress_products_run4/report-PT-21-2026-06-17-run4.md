## Executive Summary

The products-service was stress tested across 4 progressive runs (bare → DB pool + Redis cache → 4 cluster workers → 8 cluster workers). Run 4, executed with 8 workers (the Run 3 P1 recommendation), revealed a **new critical defect**: the aggregate DB connection pool (8 workers × 13 connections = 104) exceeds PostgreSQL's `max_connections` limit (~100), causing `"sorry, too many clients already"` errors at peak load. This is confirmed by Loki at 18:34:33 UTC. All 931 errors in Run 4 share this root cause — a configuration math error introduced when scaling from 4 to 8 workers. The error rate (0.34%) remains within the 0.5% SLA, but the k6 global P95 (1.57s) fails the 100ms threshold. The Black Friday 2,500 VU target requires two sequential fixes: (1) correct the DB pool math, then (2) increase workers to 12.

## Root Cause Analysis

### CRITICAL — DB Connection Pool Over-Provisioned Beyond PostgreSQL Capacity

**Observed:**
All 931 HTTP 500 errors (0.34% of requests) carry the message `"sorry, too many clients already"`. This is the PostgreSQL server-side rejection when `max_connections` is reached. It is not an application timeout — PostgreSQL refuses the connection before any query runs.

**Root cause (confirmed):**
`NODE_CLUSTER_WORKERS=8` × `DB_POOL_MAX=13` = **104 aggregate DB connections** configured by the application. PostgreSQL's default `max_connections=100` (with 3 reserved for superuser = 97 available to applications). Under peak 2,000 VU load, all 8 workers simultaneously drive toward their full connection budget, pushing the total above 100 → PostgreSQL rejects new connection requests.

**Why this was not present in Run 3 (4 workers × 25/worker = 100):**
Run 3's aggregate pool was exactly at the PostgreSQL limit (100 connections). At peak, the probability of all 4 workers simultaneously holding 25 connections was low, so only 4 errors occurred. Run 4 configured 104 connections, making exhaustion near-certain under sustained concurrent load.

**Evidence chain:**
1. Loki: `"sorry, too many clients already"` on SELECT queries to `categories`, `products`, `product_variants` — all 3 tables, confirming it's the connection layer, not a specific table/query
2. Onset at 18:34:33 UTC = t+10m54s = ~1,560 VUs in Stage 6 (1,200→2,000)
3. `nodejs_active_handles_total` peaked at 267 at 2,000 VUs (vs baseline 10)
4. No Redis errors, no OOM, no process restarts — isolates the failure to the DB connection layer
5. Error duration 482–1,078ms = time for the application to detect the connection pool failure and return 500

**What I would fix first:**
Reduce `DB_POOL_MAX` from 13 to 10 per worker: **8 × 10 = 80 total connections** (20% below the 100-connection limit, leaving headroom for superuser connections and monitoring tools). This fix takes under 5 minutes, requires no code change, and directly eliminates the source of all 931 errors. After this fix, Run 5 with 12 workers requires PostgreSQL `max_connections` to be raised to ≥ 150 (12 × 10 = 120 + headroom).

## What Happens If We Launch Now

If the Black Friday event begins with the current catalog setup, the service will handle the first ~1,500 shoppers without issues. Beyond that threshold — which represents 60% of the 2,500-shopper Black Friday target — shoppers begin seeing error pages when browsing product categories, listings, and product detail pages. They are not redirected to an alternative page — they receive a server error. At 2,000 simultaneous users, approximately 1 in 300 requests fails. At the full 2,500-user Black Friday peak, this error rate would increase meaningfully, as the root cause (database connection saturation) worsens with more users. Errors directly block shoppers from reaching the products they intend to purchase.

---

## What Needs to Happen Before Launch

- **Fix the database connection configuration** — A miscalculation when adding more server capacity means the catalog requests more database connections than the database can provide. This causes shoppers to receive errors instead of product pages. The fix is a change to two numbers in a configuration file: no code change, no deployment pipeline, under 30 minutes of engineering time.
- **Add more processing capacity** — Once the configuration is corrected, the server needs to be configured to use 12 processing workers instead of 8. This puts capacity at an estimated 20% above the Black Friday target — a comfortable safety margin.
- **Raise the database connection limit** — The database must be configured to accept slightly more simultaneous connections (150 vs the current ~100). This is a one-line configuration change to the database container.
- **Run a final validation test (Run 5)** — After the above changes, a 14-minute stress test must confirm the catalog handles 2,500 simultaneous shoppers without errors or slowdowns before Black Friday proceeds.

- **Cache expiry randomization** — A minor tuning to prevent multiple server workers from hitting the database simultaneously when a cache entry expires. The benefit is small relative to the fixes above and does not block Black Friday readiness.
